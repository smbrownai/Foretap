Foretap

A section-based iOS app launcher

What it is

Foretap replaces the App Library as the primary app-discovery surface on iPhone. Apps are organized into named sections — Pinned, Work, Travel, Reading — that stack vertically on a single screen.

Each section has its own sourcing logic: manual lists, recently used, frequently used, time-of-day windows, or Focus-mode bindings. Sections rise to the top automatically when their context applies.

Three home-screen widgets, App Intents, Shortcuts, and Siri integration are included. No subscriptions. No advertising. No tracking.

Privacy policy

Privacy is paramount

Foretap exists to help the user. The developer earns nothing from user data. There is no advertising, no analytics SDK, no third-party tracker, no telemetry pipeline, and no user account system.

What stays on the device

The following data is created and stored locally on the iPhone, in Foretap's SwiftData store and the app's App Group container. It is never transmitted:

  • Sections, their names, and the apps assigned to each.
  • Per-app usage events — when which app was tapped — used to compute recent and frequent ordering.
  • Time-window settings, Focus-mode bindings, and per-section preferences.
  • Custom URL schemes added manually for apps not in the bundled catalog.
  • Onboarding state, settings toggles, and widget snapshots.

This data is included in standard iOS device backups (iCloud or local) only when the user has enabled them. Foretap does not sync, upload, or replicate the data itself. Uninstalling the app deletes all of it. Settings → Reset erases it on demand.

The one thing that may leave the device

Foretap can optionally help build a shared, public catalog of which URL schemes open which iOS apps. This is the only network feature in the product, and it is strictly limited:

  • Trigger. A submission only happens when the user manually adds a custom app, types a URL scheme themselves, and that scheme actually launches the app. Apps picked from the bundled catalog never trigger a submission.
  • Sent. The verified URL scheme, the App Store bundle identifier, the app's display name, and an opaque per-install device hash (a SHA-256 of a random UUID generated locally on first launch). Nothing else.
  • Not sent. No identity, Apple ID, IP-derived location, installed-apps list, usage history, sections, or Focus modes.
  • Why a device hash exists. The server uses it solely to count how many distinct devices have confirmed a given scheme, and to apply a basic rate limit. It is not reversible to a person and is not joined with any other identifier.
  • Where it goes. A Cloudflare Worker operated by the developer, which writes only the fields above to a Cloudflare D1 database. Verified scheme/bundle pairs are republished as a public catalog so other Foretap users find that app pre-filled.
  • How to turn it off. Settings → Help build the catalog → off. The toggle is on by default. Disabling it is immediate and complete.

Never collected

  • No advertising identifiers (IDFA, IDFV).
  • No location, contacts, calendar, photos, or microphone access.
  • No analytics events, behavioral logs, or crash reports with personal context.
  • No third-party SDKs. The app has zero external Swift Package dependencies.
  • No accounts, no email collection, no passwords.

Children

Foretap is not directed at children under 13 and does not knowingly collect any data from them. Because the app collects no personal data from anyone, this is structurally enforced rather than policy-only.

Rights

Because no personal data is collected, there is nothing to request, export, or delete on a server. Settings → Export creates a local JSON archive of sections and preferences. Settings → Reset deletes all local Foretap data.
